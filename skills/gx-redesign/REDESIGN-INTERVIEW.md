# 개선 브리프 인터뷰 — 자산 인벤토리 + 모호함 제거 게이트

목표: 기존 자산을 전수 파악하고, 개선의 목표·범위·불가침 영역을 frontier 라운드로 확정해, 승인된 개선 브리프 하나를 만든다.

## 절차

### 1. 자산 인벤토리 (질문 금지 영역)

환경에서 확인할 수 있는 것은 전부 조사한다 — 사용자에게 묻지 않는다.

- outputs/ 전체(이전 브리프·전략·명세·final), 디자인 토큰·브랜드 가이드 문서
- .pen 디자인 파일(pencil 도구로 열람), 프로토타입 코드, 스크린샷·이미지 자산
- 서비스가 배포되어 있으면 공개 화면(WebFetch·브라우저 도구 사용 가능 시)

자산 유형별 파악 방법:

- 문서(명세·토큰·가이드): Read로 정독한다.
- .pen 파일: pencil 도구로 구조·스타일 값을 조회하고 스크린샷으로 렌더링을 확인한다.
- 프로토타입 코드: 코드를 정독하고, 실행 가능하면 실행해 실제 화면을 확인한다.
- 이미지·스크린샷: Read로 열어 시각적으로 분석한다(레이아웃·컬러·타이포그래피·위계).
- 배포된 웹 화면: 브라우저 도구로 접속해 페이지별 스크린샷을 캡처한 뒤 분석한다.

**자산이 없거나 부족하면**: 이것은 환경에서 확인할 수 없는 정보다 — 첫 라운드 질문에 자료 요청을 포함한다
(현재 화면 스크린샷 첨부, 서비스 URL, 디자인 파일 경로, Figma 링크 중 무엇을 제공할 수 있는지).
자료 없이 기억에 의한 묘사만으로 진단하지 않는다.

산출: 자산 목록 + 각 자산의 상태 요약(무엇이 있고, 언제 것이고, 어떤 체계인지).
이 인벤토리는 브리프에 그대로 들어간다.

### 2. 결정 트리 구축 (개선 전용)

- **상류**: 개선 계기(왜 지금 바꾸나 — 성과 지표 문제? 브랜드 변화? 경쟁 대응? 인상·취향?), 개선 목표와 성공 기준(무엇이 나아지면 성공인가), 개선 대상 범위(어떤 자산·화면·매체)
- **중류**: 유지 자산(불가침 — 로고·컬러·IA 등) vs 변경 허용 범위, 개선 강도 허용치(최소 수정 ~ 전면 리뉴얼), 제약(일정·개발 반영 범위·브랜드 규칙)
- **하류**: 레퍼런스·선호 톤, 매체별 세부 형식

"요즘 느낌으로"류의 모호어는 하류(취향)가 아니라 상류에서 먼저 분해한다 —
무엇이 낡았다고 느끼는지, 그래서 무엇을 달성하고 싶은지가 레퍼런스 선택보다 먼저다.

### 3~7. 라운드 규칙

[../gx-design/BRIEF-INTERVIEW.md](../gx-design/BRIEF-INTERVIEW.md)의 3~7절 규칙을 그대로 따른다:
frontier 계산(전체 질문, 의존 질문은 다음 라운드) / 라운드 실행(AskUserQuestion 4문항씩 분할, 첫 옵션 "(추천)"+근거, multiSelect·preview 활용) / 트리 갱신(총량 무제한) / 탈출구(사용자 전용, [가정] 표기) / 종료 게이트(브리프 전문 제시 → 승인 → 저장).

저장 경로: `outputs/brief/YYYY-MM-DD_<project>_redesign-brief.md`

## 개선 브리프 양식

- 프로젝트명
- 개선 대상 자산 인벤토리 (경로 포함)
- 개선 계기와 목표
- 성공 기준
- 개선 범위
- 유지 자산 (불가침)
- 변경 허용 범위
- 개선 강도 허용치
- 제약 조건
- 레퍼런스
- [가정] 목록 (탈출구 사용 시)
