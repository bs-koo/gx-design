# PROMPT-PLAYBOOK — 생성형 이미지·영상 프롬프트 규격

생성형 프롬프트 패키지(PRODUCTION-HANDOFF 경로 B)와 제작 명세의 생성형 프롬프트 절은 이 규격을 따른다.
1순위 타깃은 Gemini 웹 — "이미지 만들기"(Nano Banana 계열)·"동영상 만들기"(Omni·Veo 계열) — 이며, 타 도구 변형은 §5가 다룬다.
근거: Google 공식 프롬프팅 가이드(Veo 3.1·Nano Banana) + 검증된 커뮤니티 관례(출처는 설계 스펙 §11). 커뮤니티발 미확정 기법은 [미검증] 표기.

## 1. Gemini 웹 사용 명세 (2026-07 UI 기준)

- **진입**: Gemini 웹·앱 입력창의 "+" 메뉴 → "이미지 만들기" 또는 "동영상 만들기". Pro 모델 권장. (메뉴명이 바뀌어도 프롬프트 본문은 그대로 유효하다.)
- **템플릿**: 기본은 선택하지 않는다 — 스타일은 프롬프트 본문이 전담한다. 템플릿(단색·시네마틱 등)은 브랜드 무드와 정확히 일치할 때만 지정하고, 지정했다면 패키지에 템플릿명을 명시한다. 템플릿 목록은 UI 업데이트로 바뀔 수 있다 — 이름이 사라졌으면 미선택으로 진행한다.
- **생성 설정값이 정본**: 종횡비·해상도·클립 길이는 프롬프트 문장이 아니라 **UI 설정**(동영상 종횡비 토글 등)으로 지정한다. 프롬프트 문장 속 비율 서술("16:9 wide")은 보조 강조일 뿐 설정을 대체하지 못한다. 컷별 항목의 "생성 설정값" 필드에 별도 기재한다.
- **생성 후**: 이미지는 컷당 3~5장, 영상은 1~2회 생성 → 검수 체크리스트로 선별 → 미달 항목은 **후속 수정 프롬프트**(§2.6)로 같은 대화에서 부분 수정. 재롤(전체 재생성)은 즉시 탈락 기준 위반 시에만.

## 2. 이미지 프롬프트 규격 (Nano Banana 계열)

### 2.1 5요소 뼈대

모든 프롬프트는 아래 5요소를 서술형 문단으로 녹인다(순서 고정 아님, 누락 금지):

1. **Subject** — 무엇을: 구체 명사 + 재질·질감 수식
2. **Action** — 무슨 일이 일어나는가(정물이면 상태 서술)
3. **Location·Context** — 어디서, 어떤 환경·시간대
4. **Composition** — 구도·카메라: 샷 크기, 앵글, 렌즈감(shallow depth of field 등), 여백·오버레이 공간
5. **Style** — 매체·룩: 사진/일러스트/3D, 조명, 색, 무드

### 2.2 구체화 원칙

- 재질·텍스처는 일반명사 대신 소재+질감으로: "armor" → "ornate elven plate armor, etched with silver leaf patterns".
- 조명은 방향·질감까지: "soft window light from the left, gentle falloff".
- 뭉뚱그린 형용사(beautiful, high quality)는 정보가 없다 — 지우고 구체 서술로 바꾼다.

### 2.3 시리즈 일관성 문단 (공통 스타일 블록)

같은 프로젝트의 여러 컷은 공통 문단 하나를 모든 프롬프트 끝에 반복한다:

- 팔레트(hex 포함), 조명 톤, 카메라·필름 룩, 무드 키워드를 한 문단으로 고정
- **디자인 톤 기준([DESIGN-TONE-ANCHOR.md](DESIGN-TONE-ANCHOR.md))이 있으면** 그 파일의 팔레트 hex·폰트·톤 키워드를 이 블록의 정본 소스로 쓴다 — 임의 색 지정 금지.
- 마지막 문장: "All images in this series should share this consistent look."
- 생성 도구는 프롬프트 간 맥락을 유지하지 않는다 — 반복 포함이 유일한 일관성 장치다(같은 대화의 후속 수정은 예외).

### 2.4 네거티브 처리

- 금지 요소는 **긍정 서술로 치환**을 우선한다: "no buildings" → "a desolate, empty landscape of bare rock".
- 치환이 불가능한 것만 최소 나열(마스코트·특정 색 등). 나열식 금지어 문단을 프롬프트의 중심으로 만들지 않는다.

### 2.5 참조 이미지

- 참조를 쓰면 역할을 문장으로 고정한다: "Use Image A for the character's pose, Image B for the art style, Image C for the background environment."
- 캐릭터·심볼 일관성이 필요하면 유지 대상 특징을 명시한다: "maintain the exact same facial features — same eyes, nose shape, jawline".

### 2.6 후속 수정 프롬프트 (컷마다 2~3개 필수)

Nano Banana 계열의 최대 강점은 같은 대화에서의 부분 수정이다. 각 컷에 예상 실패 지점을 겨냥한 후속 수정 프롬프트를 2~3개 포함한다:

- **델타 지시**: 바꿀 것 하나만 말한다 — "Change the amber glow to appear in only one place."
- **유지 잠금**: "Keep everything else exactly the same, especially the composition and the palette."
- 전형 용도: 색·조명 보정 / 요소 제거·개수 교정 / 여백 확보(오버레이 자리) / 텍스트 철자 교정

## 3. 영상 프롬프트 규격 (Omni·Veo 계열)

### 3.1 5요소 뼈대

[Cinematography] + [Subject] + [Action] + [Context] + [Style & Ambiance]를 서술형으로 조합한다.

- Cinematography 어휘: 샷(wide/medium/close-up), 무브(static, slow dolly-in, tracking, crane, pan, POV), 렌즈감(35mm, shallow depth of field, wide-angle, macro)
- 멀티샷이 필요하면 타임스탬프로 나눈다: `[00:00-00:02] ...` `[00:02-00:05] ...`

### 3.2 오디오 지시 (전 클립 필수)

오디오를 명시하지 않으면 원치 않는 소리(방청객 웃음 등)가 생성될 수 있다 — 모든 클립에 오디오 지시를 넣는다:

- 대사: 자연어 화자 + 큰따옴표 — `A woman says, "We have to leave now."` 감정은 부사구로: `in a weary voice,`
- 효과음: `SFX: keyboard clicks, distant thunder`
- 배경음: `Ambient noise: quiet office hum`
- 무음 의도면 잠근다: `Ambient noise: near-silent room tone, no music, no audience sounds`
- [미검증] 대사를 콜론 형식(`Character says: "..."`)으로 쓰면 화면 자막 노출이 줄어든다는 커뮤니티 보고가 있다 — 자막이 껴서 나오면 콜론 형식으로 재시도한다.

### 3.3 길이·밀도 규율

- 클립은 8초 한계 기준으로 분할한다(4/6/8초).
- **1클립 1주제 1액션.** 프롬프트는 150~175단어를 넘기지 않는다 — 넘으면 모델이 지시를 선별적으로 무시하기 시작한다.
- 대사는 클립 길이당 중간 길이 문장 1~2개.
- [미검증] POV·셀피 샷은 문장 끝에 "(that's where the camera is)"로 카메라 위치를 못박으면 성공률이 오른다는 보고가 있다.

## 4. 온스크린 텍스트 정책

| 텍스트 유형 | 정책 |
|---|---|
| 정밀 로고·워드마크 | 프롬프트 금지 — 코드 렌더링 전담, 생성 이미지 위 합성 |
| 긴 카피·문장(영문 25자+, 한글 문장) | 프롬프트 금지 — 편집 도구 후삽입 |
| 짧은 영문 단어·숫자(1~10자) | 허용 — 따옴표로 문구 명시 + 폰트·굵기·색·위치를 문장으로: "The headline 'LAUNCH' rendered in bold white sans-serif at the top". 실측(비공식): 1~10자 정확도 약 98%, 25자 초과 시 60%대로 급락 |
| 한글 짧은 단어(8~10자 이내) | [미검증] 시도 가능 — 생성 후 철자 검수 필수, 1회 수정 실패 시 후삽입 폴백 |

- 텍스트가 이미지의 주체(포스터 헤드라인 등)면 **2단계**로 간다: ① 카피 문구 확정 → ② 그 문구의 정확한 렌더를 요구하는 프롬프트 작성.
- 텍스트를 넣지 않는 컷은 §2.4 원칙대로 긍정 서술로 잠근다: "no readable text" 대신 "all detail abstract and soft, nothing reads as text".

## 5. 타 도구 변형

- **모델 무관 서술형**: §2·§3의 서술형 본문은 특정 문법 없이 대부분의 웹 UI 도구(DALL-E, Flux 등)에 그대로 쓸 수 있다. 타 도구용 본문이 따로 필요하면 설정값 서술(비율 등)만 문장으로 복원해 제공한다.
- **파라미터형 도구(Midjourney 등)**: 5요소 본문은 유지하되 ① 비율·스타일 강도는 도구 파라미터로 분리(예: `--ar 16:9`), ② 네거티브는 도구 전용 필드 사용, ③ 시리즈 일관성 문단은 스타일 참조 기능(sref 등)으로 대체 가능함을 안내한다.

## 6. 패키지 산출 양식

저장: `outputs/final/YYYY-MM-DD_<project>_gemini-prompts.md`

**동일 산출물 A/B 원칙**: 코드 렌더링 경로와 병행 선택되면 패키지의 첫 컷(컷 0)은 **같은 명세 산출물의 동일 장면 재현 시도**여야 한다 — 사용자가 두 경로를 함께 고른 목적은 같은 산출물의 두 결과를 비교해 택1하는 것이다. 무드·보완 컷만으로 패키지를 채우면 그 비교가 성립하지 않는다. 짧은 영문 라벨은 §4 허용 범위에서 요청하고, 재현 한계(라벨 철자 불안정·좌표 근사·정본 심볼 재현 불가)와 "철자 교정 후속 수정 N회 실패 시 코드 렌더본 채택" 탈출 조건을 컷에 명기한다. 무드·보완 컷은 그 뒤에 배치한다.

1. **사용법** — 표(컷 | 용도 | 진입 메뉴 | 템플릿 | 생성 설정값) + 생성 후 처리 흐름(§1). 템플릿 열의 기본값은 "미선택".
2. **공통 스타일 블록** — §2.3 시리즈 일관성 문단(복붙용 코드블록).
3. **컷별 항목** — 각 컷마다: 목적(한국어 1줄, 명세의 절·시간 구간 인용) / 생성 설정값(UI로 지정할 값) / PROMPT(복붙용, 5요소 + 공통 블록 포함) / 후속 수정 프롬프트 2~3개(§2.6) / 한국어 해설 / 검수 포인트 / 실패 시 변주 1개(단순화 방향).
4. **합본 가이드** — 로고 합성·타이틀 조판·클립 연결 규칙(해당 시).
5. **검수 체크리스트** — 즉시 탈락 기준(브랜드 금지 요소·텍스트 오렌더 등) + QA 4항목(AI 아티팩트 / 기술 정확성 / 구성 문제 / 스타일 일관성) + 컷별 검수 포인트 통과.
