# gx-design v0.4.0 설계 스펙 — 실물 품질·프롬프트 규격·모드 분기

- **작성일**: 2026-07-24
- **대상 버전**: v0.3.0 → v0.4.0
- **승인 경로**: 브레인스토밍 인터뷰(생성 경로·핵심 모드 수준·Gemini 환경 확인) → 접근안 A 선택 → 생태계/프롬프트 가이드/MCP 지형 조사 3건 반영 → 사용자 최종 승인

---

## 1. 배경과 문제

v0.3.0 파이프라인을 두 프로젝트(reels 영상, bx 브랜드)로 완주한 결과 세 가지 문제가 확인됐다.

1. **실물 품질이 문서 품질을 못 따라간다.** 명세 md는 프레임 단위 검산까지 있는 400~500줄 수준인데, 코드 렌더링 실물(심볼·키 비주얼)은 조잡하다. 직접 원인: 렌더 결과 이미지를 아무도 눈으로 보고 비평하지 않는다 — "명세 값 대조"만 있고 시각 품질 루프가 없다.
2. **Gemini 프롬프트 패키지가 실제 사용 환경과 어긋난다.** 팀은 Gemini 웹 채팅의 "이미지 만들기"(Nano Banana 2)·"동영상 만들기"(Omni)에 붙여넣어 쓴다. 현행 패키지는 (a) 템플릿 선택 개념이 없고, (b) 원샷 생성+재롤만 가정해 Nano Banana의 대화형 부분 수정 강점을 버리며, (c) 종횡비를 프롬프트 문장에 섞고(정본은 UI/파라미터 설정), (d) 네거티브를 "Avoid entirely:" 나열로 쓰고(공식 권장은 긍정 서술 치환), (e) 텍스트 전면 금지 원칙이 최신 모델의 실측 성능(영문 1~10자 정확도 ~98%)보다 과도하게 보수적이다.
3. **1회 실행에 약 2시간 걸린다.** 원인: 브리프 인터뷰 라운드 무제한, 리서치 축별 병렬 + 300~460줄 리포트, 전략가 3개 병렬(각 130~210줄), 검수 2축 병렬 + 재검수 루프, 그리고 단계별 산출물 분량 하한이 없어 매번 최대치로 쓰는 구조.

### 1.1 조사 결론 (2026-07-24, 서브에이전트 3건)

- **Claude Code 생태계**: "붙여넣기용 프롬프트 md 패키지"를 만드는 완성형 스킬·플러그인은 없음. 대부분 API 직접 호출형(실행 도구). 차용 가능한 검증된 부품: Google 공식 5요소 공식, QA 체크리스트 4항목, 계층 구조(공통 규칙→모델별→검증), multi-turn 편집 시 "유지할 요소 명시" 규칙.
- **프롬프트 모범 사례**: Google 공식 가이드(Veo 3.1 / Nano Banana / Gemini API 문서)에서 프롬프트 구조·오디오 문법·텍스트 렌더링 한계·종횡비 정본이 확정됨. §4에 반영.
- **MCP 지형**: 이미지·영상 생성 MCP는 전반적으로 초기 단계. 이번 버전에서는 도입하지 않고 §9(향후 확장)에 후보만 기록.

## 2. 목표

1. 코드 렌더링 실물에 **시각 품질 루프**를 의무화해 "조잡한 실물"을 구조적으로 차단한다.
2. 생성형 프롬프트 패키지를 **Gemini 웹 실사용 환경(Nano Banana 2 / Omni) 기준**으로 재규격화하고, 타 도구 변형도 함께 제공한다.
3. **전체/핵심 모드**를 도입해 핵심 모드에서 게이트를 유지한 채 내부를 경량화한다(순수 실행 시간 목표 30분 내외).
4. 제작 명세 md에 **매체별 필수 섹션과 완성 판정 기준**을 부여한다.

### 2.1 비목표 (범위 외)

- 이미지·영상 생성 MCP 통합(지형만 기록, §9).
- 기존 bx/reels 산출물 재생성(스킬 개선 후 별도 실행으로 검증 겸 수행 가능).
- 오케스트레이터 구조 재편(스킬 분리·에이전트 증설) — 접근안 B·C로 검토 후 기각.
- README·구축 가이드 문서 전면 개정(버전 표기·신규 파일 언급 수준의 최소 갱신만 포함).

## 3. 설계 1 — 전체/핵심 모드 분기

### 3.1 모드 결정 규칙

- 스킬 인자에 모드가 있으면 그대로 사용한다(예: `/gx-design 핵심 사내 발표용 키 비주얼`). 인식 토큰: "핵심"·"빠르게"·"경량"(핵심 모드), "전체"·"풀"(전체 모드).
- 인자에 없으면 **브리프 인터뷰 첫 라운드의 첫 문항**으로 모드를 묻는다(AskUserQuestion, header "진행 모드", 옵션 2개, 상황에 맞는 쪽에 "(추천)" 표기 — 산출물 범위가 넓거나 이해관계자가 많으면 전체, 단일 산출물·마감 촉박이면 핵심).
- 모드는 브리프에 기록하고, 이후 모든 단계 보고에 현재 모드를 표기한다.
- **모드 전환**: 사용자가 도중에 요구하면 전환한다. 핵심→전체 전환 시 이미 통과한 게이트는 다시 열지 않고 남은 단계만 전체 규칙을 적용한다.

### 3.2 핵심 모드 단계 규칙 (gx-design)

게이트 3개(브리프 승인·전략 선택·실물 제작)는 전부 유지한다. 내부만 경량화한다.

| 단계 | 전체 모드(현행 유지) | 핵심 모드 |
|---|---|---|
| 0 브리프 | 라운드 무제한(frontier 소진까지) | **1라운드(최대 4문항, 상류 결정 우선)**. 미해소 결정은 [가정] 표기로 채움. 승인 게이트 동일 |
| 1 리서치 | 조사 축별 병렬 스폰, 리포트 복수 | **design-researcher 1개, 통합 리포트 1개, 상한 100줄**. 웹 조사는 결정에 직접 필요한 것만 |
| 2 전략 | strategist 3개 병렬(3안) + 비교표 | **strategist 1개가 주 콘셉트 1안(콘셉트 형식 준수) + 반대 방향 미니 대안(10줄 요약)** 작성. 선택 게이트 유지 — 옵션: 주안 채택 / 대안 확장(strategist 재위임) / 방향 수정 |
| 3 제작 | 분량 제한 없음 | **매체별 필수 섹션만, 상한 200줄**(§6 필수 섹션 체크리스트 기준) |
| 4 검수 | 2축 병렬 + 축별 재검수 | **검수 에이전트 1개가 축 A→축 B 순차 점검**. 보고는 축별 분리 유지(합산 금지 원칙 계승). Critical만 수정 루프, Major 이하는 기록만 |
| 5 실물 | 게이트 + 제작 | **동일 — 경량화하지 않는다**(품질 개선의 핵심 지점). VISUAL-QUALITY 루프도 동일 적용 |

### 3.3 핵심 모드 단계 규칙 (gx-redesign)

같은 원칙을 적용한다: 단계 1 진단은 검수 에이전트 1개 통합(보고는 축별 분리), 단계 2 갭 리서치는 기본 생략(브리프가 경쟁·트렌드 목표를 명시한 경우에만 경량 실행), 단계 3 전략은 1안(개선 강도는 브리프의 허용치 기준으로 선택) + 미니 대안, 단계 5 검수는 1개 에이전트가 축 A→B→C 순차 점검. before/after 매핑 의무는 모드와 무관하게 유지.

### 3.4 문서 반영 위치

- `skills/gx-design/SKILL.md`: 단계 0 앞에 "모드 결정" 절 신설 + 각 단계에 핵심 모드 규칙 1~2줄 병기.
- `skills/gx-redesign/SKILL.md`: 동일 방식.
- `skills/gx-design/BRIEF-INTERVIEW.md`: 핵심 모드일 때 라운드 규칙 대체 조항(1라운드 상한) 추가.
- 합리화 표에 추가: "핵심 모드니까 게이트도 줄이자" → "핵심 모드가 줄이는 것은 내부 작업량이다. 게이트는 모드와 무관하게 유지된다."

## 4. 설계 2 — PROMPT-PLAYBOOK.md 신설 (생성형 프롬프트 규격)

위치: `skills/gx-design/PROMPT-PLAYBOOK.md`. PRODUCTION-HANDOFF 경로 B가 이 문서를 참조하고, creative-producer 위임 프롬프트에 경로가 포함된다. gx-redesign도 동일 참조.

### 4.1 §1 Gemini 웹 사용 명세

- 진입 안내: "+" 메뉴 → "이미지 만들기"(Nano Banana 2) / "동영상 만들기"(Omni). Pro 모델 권장.
- **템플릿 선택 지시**: 기본은 템플릿 미선택 — 스타일은 프롬프트가 전담한다. 브랜드 무드와 정확히 일치하는 템플릿이 있을 때만 명시 지정한다. 템플릿 이름(단색·시네마틱 등)은 UI 변경 가능성이 있으므로 예시로 취급한다.
- **생성 설정값 필드 분리(정본 규칙)**: 종횡비·해상도·클립 길이는 프롬프트 문장이 아니라 UI 설정/파라미터가 정본이다. 패키지의 컷별 항목에 "생성 설정값" 필드를 프롬프트 본문과 분리해 명시한다. 프롬프트 문장 내 비율 서술은 보조 강조로만 허용.

### 4.2 §2 이미지 프롬프트 규격 (Nano Banana 2)

- **뼈대 — Google 공식 5요소**: Subject / Action / Location·Context / Composition / Style.
- **재질·텍스처 구체화 원칙**: 일반명사 대신 소재+질감 서술("armor" → "ornate elven plate armor, etched with silver leaf patterns").
- **시리즈 일관성 문단**: 필름스톡·색보정·조명·카메라를 한 문단으로 고정하고 "All images in this series should share this consistent look"를 명시 — 현행 "공통 스타일 블록"을 이 형식으로 계승.
- **네거티브 처리**: 부정 나열 대신 긍정 서술 치환을 우선한다("no buildings" → "a desolate landscape"). 불가피한 금지 요소만 최소한으로 나열.
- **후속 수정 프롬프트(필수)**: 컷마다 2~3개 — 부분 수정 델타 지시("change the man's tie to green" 식) + **유지할 요소 명시**("keep everything else exactly the same, especially ...") 규칙. 재생성(재롤)은 즉시 탈락 기준 위반 시에만.
- **참조 이미지 역할 명시**: 참조를 쓸 때 "Use Image A for pose, Image B for style" 형식으로 역할을 문장으로 고정.

### 4.3 §3 영상 프롬프트 규격 (Omni / Veo 계열)

- **뼈대 — 공식 5요소**: [Cinematography] + [Subject] + [Action] + [Context] + [Style & Ambiance]. 카메라 무브·렌즈 용어집 요약 포함(dolly/tracking/crane/pan/POV, shallow DOF 등).
- **멀티샷**: 타임스탬프 표기(`[00:00-00:02] ...`) 지원. 클립은 8초 한계 기준 분할(현행 유지).
- **오디오 문법(고정 필드)**: 대사는 큰따옴표(`A woman says, "..."`), 효과음 `SFX:`, 배경음 `Ambient noise:`. **오디오 미명시 시 원치 않는 소리가 생성되는 할루시네이션 방지를 위해 모든 클립에 오디오 지시를 필수로 포함**(무음 의도면 "quiet ambiance, no audience sounds" 식으로 잠금).
- **길이 규율**: 클립당 1주제 1액션, 프롬프트 150~175단어 상한(초과 시 지시 무시 시작). 대사는 클립 길이에 맞춰 중간 문장 1~2개.
- **[미검증] 커뮤니티 팁 절**: 콜론 대사 문법(자막 노출 방지 주장), 카메라 위치 괄호 구문("(that's where the camera is)") 등은 별도 절에 [미검증] 표기로 수록하고 실패 시 A/B 시도 옵션으로만 안내.

### 4.4 §4 온스크린 텍스트 정책 (완화)

- 정밀 로고·워드마크·긴 카피·한글 문장: 여전히 코드 렌더링·후삽입 전담(현행 원칙 유지).
- **짧은 영문 단어·숫자(1~10자)**: 프롬프트 시도 허용. 따옴표로 문구를 명시하고 폰트·굵기·위치를 문장으로 서술("The headline 'URBAN EXPLORER' rendered in bold, white, sans-serif font at the top"). 실측 근거: 영문 1~10자 정확도 약 98%, 25자 초과 시 60%대(비공식 실증 자료 — 출처 명기).
- **한글**: 8~10자 이내 보수 적용, [미검증] 표기 필수 + 생성 후 철자 검수 항목 추가. 실패 시 후삽입 폴백.
- **2단계 방식**: 텍스트가 이미지의 주체인 경우 카피 문구를 먼저 확정한 뒤 렌더 지시 프롬프트를 구성한다.

### 4.5 §5 타 도구 변형

- **모델 무관 서술형**(현행 방식 계승): 특정 문법 없이 통용되는 서술형 본문 — Gemini 외 도구(DALL-E·Flux 웹 UI 등)에 그대로 사용.
- **파라미터형 도구 변환 규칙**: Midjourney 등 파라미터 문법 도구용으로 5요소를 변환하는 규칙(종횡비·스타일 파라미터 분리 등)을 간결히 수록.

### 4.6 §6 패키지 산출 양식

기존 골격(사용법 / 공통 스타일 블록 / 컷별 항목 / 합본 가이드 / 검수 체크리스트)을 계승하되:

- 사용법 표에 **메뉴·템플릿·생성 설정값(종횡비/해상도/길이)** 열 추가.
- 컷별 항목 = 목적 / **생성 설정값** / PROMPT(5요소 구조) / **후속 수정 프롬프트 2~3개** / 한국어 해설 / 검수 포인트 / 실패 시 변주 1개.
- 검수 체크리스트에 **QA 4항목**(AI 아티팩트 / 기술 정확성 / 구성 문제 / 스타일 일관성) 통합.
- 저장 경로 현행 유지: `outputs/final/YYYY-MM-DD_<project>_gemini-prompts.md`.

### 4.7 PRODUCTION-HANDOFF.md 수정

- 경로 B 절의 패키지 구조·원칙 서술을 PROMPT-PLAYBOOK 참조로 대체(중복 서술 제거, 핵심 원칙 2~3줄 요약만 유지).
- "온스크린 텍스트는 프롬프트에 넣지 않는다" 문장을 §4.4의 완화 정책 참조로 교체.

## 5. 설계 3 — VISUAL-QUALITY.md 신설 (코드 렌더링 시각 품질 루프)

위치: `skills/gx-design/VISUAL-QUALITY.md`. PRODUCTION-HANDOFF 경로 A(코드 렌더링)와 UI 프로토타입 경로에 의무 적용.

1. **제작 전**: frontend-design 스킬이 가용하면 로드한다(심미 기준 확보).
2. **렌더 후 시각 비평(의무)**: 결과 이미지(스틸·PNG·스크린샷)를 Read로 직접 열어 본다. 비평 기준:
   - 디자인 품질: 시각 위계, 간격·정렬 일관성, 밀도(빈약/과밀), 색 균형·대비
   - **정체성 오독 체크**: 산출물이 의도와 다른 것으로 읽히지 않는가(예: 심볼이 막대 차트로, 키 비주얼이 와이어프레임으로)
   - QA 4항목: AI 아티팩트(렌더 깨짐·폰트 폴백 흔적) / 기술 정확성(규격·해상도) / 구성 문제 / 스타일 일관성(다른 자산과)
3. **루프**: 비평 → 수정 → 재렌더. **최소 1회, 최대 3회.** 회차별 비평 요지를 실물 옆 기록(제작 문서 또는 렌더 로그 절)에 남긴다. 3회 후에도 Critical이 남으면 사용자에게 한계와 대안(경로 변경·수동 보정)을 보고한다.
4. **통과 기준**: 자체 비평 Critical 0건.
5. **검수 연계**: 전체 모드의 검수 축 B 입력에 실물 이미지 경로를 포함한다 — 문서만 검수하던 맹점 제거. (핵심 모드 통합 검수도 동일.)
6. 대표 컷 스틸 우선 렌더 원칙(현행)은 유지 — 루프는 대표 컷에서 먼저 돌리고 전체 렌더는 통과 후 수행.

## 6. 설계 4 — 제작 명세 md 양식 강화

`skills/creative-production/SKILL.md`에 추가:

- **매체별 필수 섹션 체크리스트**: 현행 "출력 항목" 나열을 완성 판정 기준으로 승격 — 각 매체(BX/그래픽/UX·UI/영상)에 대해 "이 섹션이 없으면 미완성" 목록을 명시하고, 자체 검수 절에서 체크한다.
- **생성형 프롬프트 규격 준수**: 명세에 포함되는 생성형 AI 프롬프트는 PROMPT-PLAYBOOK 규격(5요소·설정값 분리·후속 수정)을 따라야 한다는 완료 기준 추가. 기존 "대상, 구성, 스타일, 조명, 재질, 카메라, 비율, 제외 요소" 나열 원칙은 PLAYBOOK 참조로 대체.
- **핵심 모드 분량 상한**: 위임 프롬프트에 상한이 명시되면 따른다는 조항(상한 자체는 오케스트레이터가 §3.2 표 기준으로 전달).

## 8. 엣지 케이스와 에러 처리

- **모드 미지정 + 인터뷰 탈출구 발동**: 탈출구("그만, 진행해")가 모드 문항 응답 전에 발동되면 핵심 모드로 간주하고 [가정] 표기한다(탈출구의 취지가 속도이므로).
- **핵심 모드에서 리서치 부족 판명**: 전략 단계에서 근거 부족이 드러나면 오케스트레이터가 보충 리서치 1회를 스폰할 수 있다(상한 100줄 동일). 무단 전체 모드 전환은 금지.
- **Gemini UI 변경**: 메뉴 이름·템플릿 목록이 바뀌어도 프롬프트 본문은 유효하도록, UI 안내와 프롬프트 본문을 분리 서술한다. UI 안내에는 "2026-07 기준" 표기.
- **VISUAL-QUALITY 3회 초과**: Critical 잔존 시 자동 진행하지 않고 사용자 보고 + 경로 대안 제시.
- **한글 텍스트 생성 실패**: 검수에서 철자 오류 발견 시 후삽입 폴백을 자동 제안(재생성 무한 루프 방지).

## 9. 향후 확장 (이번 범위 외 기록)

파이프라인 내 자동 생성(생성→검수→재생성 루프)을 붙일 경우 후보(2026-07 조사 기준, 전반적으로 초기 단계):

- **nanobanana-mcp-server** (zhongweili, ⭐381) — Gemini API 키만으로 시작, 이미지 전용, Windows 지원. 커뮤니티 중 가장 성숙.
- **Replicate 공식 MCP** — 회사 공식 배포로 신뢰도 최고, 다중 모델 허브.
- **Google mcp-genmedia** (GoogleCloudPlatform, experimental) — Imagen+Veo+음성까지 유일하게 전체 커버, 단 GCP 프로젝트·ADC 셋업 필요.
- **fal.ai MCP** (⭐52) — 벤더 락인 없는 600+ 모델 허브.

PROMPT-PLAYBOOK의 규격(5요소·설정값 분리)은 API 파라미터와 1:1 대응되므로 MCP 도입 시 그대로 재사용 가능하다.

## 10. 검증 방법

1. **정합성 점검**: 수정된 문서 간 상호 참조(SKILL ↔ 보조 문서 ↔ 에이전트)가 깨지지 않았는지 링크·경로 확인.
2. **드라이런(핵심 모드)**: 소형 과제(예: 플러그인 소개 SNS 이미지 1종)로 핵심 모드 전 단계를 실행해 (a) 인터뷰 1라운드 준수, (b) 산출물 분량 상한 준수, (c) 게이트 3개 작동, (d) VISUAL-QUALITY 루프 기록 존재를 확인.
3. **프롬프트 패키지 실사용 검증**: 산출된 gemini-prompts.md를 실제 Gemini 웹에 붙여넣어 이미지 1종 생성(사용자 수행) — 후속 수정 프롬프트가 실제로 작동하는지 확인.
