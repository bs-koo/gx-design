# 디자인 톤 기준(Design Tone Anchor) 통합 설계

- **일자**: 2026-07-24
- **대상**: gx-design / gx-redesign 오케스트레이터
- **상태**: 승인됨(방향 승인 후 스펙화) — 브랜치 `feat/design-tone-anchor`
- **버전 영향**: 0.4.0 → 0.5.0 (사용자향 신규 능력 추가 = minor)

## 목표(한 문장)

사용자가 첨부한 디자인 시스템 파일(DESIGN.md 등)을 **디자인 톤 기준**으로 받아, 팔레트·타이포·톤·형태를 전략→제작→검수 전 단계에 구속력 있게 물려주는 경로를 추가한다.

## 배경 / 문제

- 첫 드라이런에서 드러난 문제: 색·타이포가 매 단계 즉흥적으로 정해져 "조잡"해진다.
- 현재 브리프의 "사실 수집"은 *사용자가 준 자료*를 읽긴 하지만, **디자인 톤 기준을 명시적으로 묻지도, 하류에 구속으로 물려주지도 않는다.** 첨부되어도 참고자료 한 줄로 흘러간다.
- `DESIGN.md`는 Google Labs 공식 포맷(YAML frontmatter 토큰 + 마크다운 근거: Colors/Typography/Layout·Spacing/Elevation/Shapes/Components/Do's·Don'ts). designmd.co는 그 포맷의 461개+ 기성 카탈로그 + MCP.

## 접근 (승인된 방식 = "첨부 기반, 자동 당김 아님")

- 파이프라인이 카탈로그를 **자동으로 당겨오지 않는다.** MCP를 **의존하지 않는다.**
- 대신 브리프/자산 단계에서 **"디자인 톤 기준 파일 있냐"고 명시적으로 묻고**, 첨부받으면 그것을 톤 기준으로 삼는다.
- 파일이 없으면 designmd.co 카탈로그를 *"골라서 첨부할 출발점"* 으로만 **안내**(soft pointer). 외부 의존 0, 브랜드 오염 위험 0.

이 방식이 대안(카탈로그 MCP 자동 검색·당김)보다 나은 이유: (1) 외부 MCP 미설치 환경에서도 안 깨짐, (2) 카탈로그 기성 파일이 전략 게이트를 우회해 브랜드를 제네릭하게 만드는 위험을 회피, (3) 기존 자산 파악 흐름과 결이 일치.

## 핵심 개념 — 톤 기준의 구속 강도

| 강도 | 정의 | 무엇이 되나 | 판정 기준 |
|---|---|---|---|
| **hard(구속)** | 사용자 자신의 브랜드/디자인 시스템 파일 | 팔레트·타이포·톤·형태(radius)가 **하드 제약**. 전략은 콘셉트·구조만 그 안에서 탐색. 이탈은 근거 명시 + 전략 게이트에서 사용자 승인 필요 | 검수에서 근거 없는 이탈 = 결함 |
| **soft(참고)** | 카탈로그에서 고른 출발점, 열망 레퍼런스 | 톤을 **안내**. 전략이 이탈 가능하되 근거를 기록 | 검수에서 이탈은 근거 기록만 확인 |

- **강도는 첨부 시 되물어 확정**한다. 기본 규칙: *사용자 자신의 브랜드 파일 = hard*, *카탈로그/열망 레퍼런스 = soft*.

## 파이프라인 배선

### ① 인테이크 (단계 0) — 두 스킬 공통

- **gx-design BRIEF-INTERVIEW.md**: "사실 수집"에서 톤 기준 파일을 능동 탐색하고, 없으면 첫 라운드에 **중류 제약 질문**으로 묻는다(하류 스타일 질문 아님 — "포지셔닝 전 컬러 3안 금지" 규칙 위반 아님).
- **gx-redesign REDESIGN-INTERVIEW.md**: "자산 인벤토리"에 톤 기준 파일을 포함시키고, 없으면 자료 요청 라운드에 함께 요청한다.
- 첨부 시: 파일을 정독→**팔레트(hex)·타이포·톤 키워드·형태(radius) 요약 추출**→브리프에 **`디자인 톤 기준`** 항목으로 강도와 함께 박는다. 원본 파일 경로는 제작이 그대로 인용할 정본으로 보존.
- 파싱: DESIGN.md는 마크다운이라 **Read로 충분**. frontmatter 토큰이 있으면 그것을, 없으면 본문/브랜드 가이드 산문에서 추출. 별도 코드 파서 없음.

### ② 전략 (gx-design 단계 2 / gx-redesign 단계 3)

- 톤 기준이 있으면 팔레트·타이포·톤·형태는 **탐색 축이 아니라 제약**.
- Design-It-Twice(및 redesign의 3안)는 콘셉트·구조를 그 제약 안에서 탐색.
- hard: 이탈 시 근거 + 전략 게이트 승인. soft: 이탈 허용하되 근거 기록.

### ③ 제작 (gx-design 단계 3 / gx-redesign 단계 4)

- hex·폰트·spacing·radius를 톤 기준 파일에서 **그대로** 가져와 명세에 박는다. ← "색 조합 조잡함"을 직접 해결하는 지점.
- **보너스**: 그 팔레트를 PROMPT-PLAYBOOK §2.3 공통 스타일 블록의 hex 소스로 주입 → Gemini 이미지도 같은 톤.

### ④ 검수 (gx-design 단계 4 / gx-redesign 단계 5)

- **톤 기준 준수 점검**을 축 A(전략 일치) 입력에 추가: 산출물이 톤 기준 토큰을 지켰는가.
- hard: 근거 없는 이탈 = 결함. soft: 이탈이 근거와 함께 기록됐는지 확인.

## 규칙 중앙화

규칙을 파일마다 복붙하지 않는다. **신규 참조 파일 `skills/gx-design/DESIGN-TONE-ANCHOR.md`** 하나에 정의(톤 기준 정의 / 인테이크·파싱 / 구속 강도 / 단계별 흐름 / designmd.co 안내 / 부재 시 우아한 동작)하고, 나머지 파일은 이를 링크로 참조한다(PROMPT-PLAYBOOK·VISUAL-QUALITY와 동일 패턴). gx-redesign은 `../gx-design/DESIGN-TONE-ANCHOR.md`로 교차 참조(기존 관례).
